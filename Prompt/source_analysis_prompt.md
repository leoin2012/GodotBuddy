## 分析要求

**核心原则：以 UE 为锚点讲解 Godot。** 读者已经熟悉 UE，所以每个 Godot 概念都要先说"这相当于 UE 的什么"，再讲"但 Godot 的做法有何不同"。

请按照以下结构生成分析报告：

### 1. 模块概览 — "UE 程序员 30 秒速览"
- 用一句话说明：这个模块在 Godot 中做什么，对应 UE 的哪个模块
- 核心类/结构体列表，每个都标注 UE 对应物
- 一张 **Godot vs UE 概念速查表**（表格形式）

### 2. 架构对比 — "同一个问题，两种解法"
- Godot 的架构设计（用 mermaid 类图展示）
- UE 对应模块的架构设计（简要描述）
- **关键架构差异分析**：
  - 设计哲学差异（如：Godot 的"一切皆节点" vs UE 的"Actor-Component"）
  - 继承体系差异
  - 模块耦合方式差异

### 3. 核心实现对比 — "代码层面的差异"
对每个核心功能点：
- **Godot 怎么做的**：关键代码片段 + 执行流程
- **UE 怎么做的**：对应的 UE 实现方式（引用 UE 源码路径和关键类名）
- **差异点评**：哪种更好？各自的 trade-off 是什么？
- 涵盖：内存管理策略、线程安全、数据结构选择、算法实现

### 4. UE → Godot 迁移指南
- **思维转换清单**：UE 程序员需要"忘掉"什么、"重新学"什么
- **API 映射表**：常用 UE API → Godot 等价 API（表格形式）
- **陷阱与误区**：UE 程序员最容易踩的坑
- **最佳实践**：在 Godot 中如何用"Godot 的方式"做事

### 5. 性能对比
- Godot 该模块的性能特征
- 与 UE 对应模块的性能差异
- 对于性能敏感场景的建议

### 6. 总结 — "一句话记住"
- 用一句话概括该模块 Godot vs UE 的核心差异
- 设计亮点（Godot 做得比 UE 好的地方）
- 设计短板（Godot 不如 UE 的地方）
- UE 程序员的学习路径建议
